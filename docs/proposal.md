# Proposal：夸克PC桌面端Mini知识助手

## 1. Problem Statement

### 1.1 产品方向分析

夸克产品矩阵当前有三个主要方向：**夸克网盘**（文件存储与协作）、**千问**（AI对话）、**夸克浏览器**（搜索入口）。

#### 候选功能方向

**方向A：AI网盘知识助手**
> 以夸克网盘为数据底座，为存储在网盘中的文档/图片/PDF提供AI语义搜索、智能摘要、文件知识图谱。用户对着自己的文件"提问"，AI理解内容后回答。

| 维度 | 评估 |
|------|------|
| 可行性 | 中高。网盘提供文件列表和预览API，AI分析层可复用千问能力。Mock场景下可用本地文件模拟 |
| 展示价值 | **高**。可视化强——文件列表 + AI分析结果对比，评审一眼能看懂价值。且"网盘+AI"是行业热点，故事线清晰 |
| 面试加分点 | 同时展示夸克两大产品的协同能力，覆盖面广 |

**方向B：千问桌面快捷助手**
> 一个桌面常驻迷你窗口，悬浮/侧栏形态，快速发起AI对话。支持划词翻译、选中文本回答、多轮对话历史。

| 维度 | 评估 |
|------|------|
| 可行性 | 高。本质是一个封装了千问API的聊天UI，技术复杂度低 |
| 展示价值 | 中。演示效果直观但"又一个ChatGPT套壳"的风险高，缺乏差异化记忆点 |
| 面试加分点 | 低——技术深度不够，难以展示架构能力 |

**方向C：浏览器搜索增强（聚合知识搜索）**
> 在搜索框输入query后，同时返回网页搜索结果 + 网盘相关文件 + AI生成的知识摘要卡片。

| 维度 | 评估 |
|------|------|
| 可行性 | 低。需要同时mock搜索、网盘、AI三套服务，复杂度高且边界模糊 |
| 展示价值 | 中上。概念好但Mini Demo很难在有限时间内做出"完整感" |
| 面试加分点 | 如果做出来会很亮眼，但失败风险也最大 |

### 1.2 推荐方向

**推荐方向A：AI网盘知识助手。**

理由：
- 展示价值最高——文件管理界面 + AI智能分析，视觉冲击力强
- 可行性可控——核心路径清晰（文件列表 → 选择文件 → AI分析 → 展示结果），可Mock掉真实API
- 符合"知识助手"的命名，语义自洽
- 评估权重中"取舍决策"占20%，这个方向天然需要做出合理的范围取舍，能体现判断力

## 2. 技术栈候选对比

### 2.1 桌面框架

| 维度 | Electron | Tauri |
|------|----------|-------|
| **产物体积** | 大（~80-150MB，含Chromium） | 小（~5-15MB，用系统Webview） |
| **打包复杂度** | 成熟（electron-builder / electron-forge 一行命令） | 需Rust工具链 + Tauri CLI，跨平台配置略繁琐 |
| **Rust学习成本** | 无 | 有（即使只写简单IPC，调试时也需要读Rust） |
| **生态成熟度** | 极高（VS Code、Slack、Discord都在用） | 快速成长中，但社区资源、排障经验远少于Electron |
| **调试体验** | Chrome DevTools，零学习成本 | 需额外配置DevTools，部分平台有限制 |
| **CI/CD友好度** | 纯Node.js环境，任何CI都能构建 | 需要Rust环境，CI配置更复杂 |

### 2.2 前端框架

| 维度 | React | Vue 3 | Svelte |
|------|-------|-------|--------|
| **生态** | 最强（组件库、工具链、教程最多） | 强（Element Plus、Ant Design Vue等） | 中等（组件库较少，选型受限） |
| **TypeScript支持** | 优秀（本身就是TS生态的一部分） | 良好（3.3+ TS支持大幅改善） | 良好 |
| **桌面端适配** | Electron官方示例优先React | 支持但非首选 | 支持但案例少 |
| **上手速度** | 中等（Hooks心智模型需适应） | 快（模板语法直观） | 快（编译时框架，API简洁） |
| **组件库** | Ant Design / MUI / Chakra，选择丰富 | Element Plus / Naive UI，中文生态好 | 选择较少 |

### 2.3 推荐组合

**推荐：Electron + React + TypeScript + Vite**

理由：
- 评估权重中"能跑能打包"占25%，Electron的打包成熟度是最安全的选择
- "代码质量"占30%，React + TS的生态最成熟，类型定义、lint规则、测试工具链最完善
- "AI编码工具使用清醒度"占25%，Claude Code对Electron + React的代码生成最准确
- Tauri虽然产物体积小（这是它的核心优势），但Rust学习成本和打包不确定性在面试场景下风险 > 收益
- Vue也是合理选择（特别是如果团队熟悉Vue），但React的桌面端案例更多，排障成本低

> **⚠️ 待你决定**：技术栈是否接受此推荐？如果团队偏好Vue或想挑战Tauri请告知。

## 3. Scope预判

### 3.1 最小展示路径（Must Have）

1. **桌面窗口**：Electron主进程 + React渲染进程，窗口尺寸合理，有基础标题栏
2. **文件浏览面板**：左侧文件树/列表，Mock夸克网盘的文件结构（文件夹 + 文件，支持图标区分类型）
3. **AI分析面板**：右侧/下方区域，选中文件后展示AI分析结果（Mock响应即可，留真实API接入点）
4. **交互闭环**：点击文件 → 展示文件信息 → 触发AI分析 → 展示结果，全流程可操作

### 3.2 锦上添花（Nice to Have）

- 文件类型图标（文档/图片/PDF/视频等）
- AI对话式交互（底部输入框，直接对文件提问）
- 文件搜索/过滤
- 暗色主题切换
- 拖拽文件上传（模拟）
- 分析结果导出
- 多文件批量分析
- 真实的夸克网盘API对接
- 真实的千问API对接

### 3.3 明确不做

- 用户登录/认证系统
- 实时文件同步
- 数据库持久化
- 多平台打包（仅保证Windows/macOS其一可运行）
- 自动化测试（时间不够可省略，但代码结构预留测试接口）

## 4. 风险清单

### 技术风险

| 风险 | 等级 | 描述 | 应对 |
|------|------|------|------|
| Electron打包失败 | 中 | 某些依赖在特定OS版本下native模块编译失败 | 提前在目标环境验证打包，预留半天排障 |
| 类型系统膨胀 | 低 | TypeScript严格模式下大量any蔓延 | 先定义核心接口，逐步完善，允许部分TODO |
| Mock数据与真实API衔接 | 中 | Mock数据结构与真实API不一致导致重构 | Mock层按真实API返回格式设计，留注释标注 |

### 时间风险

| 风险 | 等级 | 描述 | 应对 |
|------|------|------|------|
| 范围膨胀 | **高** | 想加的功能太多，核心路径没打磨好 | 严格遵循最小展示路径优先，Nice to Have需明确排期 |
| Electron环境配置 | 中 | Node.js版本、npm依赖冲突、C++编译工具链缺失 | 用官方starter template起步，不自己从零搭 |
| 设计耗时过长 | 中 | 在UI细节上花太多时间 | 用现成组件库（Ant Design / MUI），不手写样式 |

### 打包风险

| 风险 | 等级 | 描述 | 应对 |
|------|------|------|------|
| 产物过大 | 低 | Electron + 依赖导致安装包臃肿 | 用electron-builder的asar打包，排除devDependencies |
| 跨平台不一致 | 中 | 开发环境能跑，打包后白屏或IPC失败 | 开发阶段就用`electron .`验证，最后再`electron-builder`打包 |
| 依赖遗漏 | 低 | devDependencies和dependencies混淆 | 按规范分类，打包前用`npm ci`验证 |
